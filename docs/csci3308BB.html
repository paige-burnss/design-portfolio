CSCI 3308: Application Development Project


In Spring 2022, I took a class called CSCI 3308: Software Development Methods and Tools in the Engineering School at CU Boulder. In this class I learned front-end design and construction as well as back-end database design and construction, and full-stack integration.

As a part of this class, I was also put into a team with 4 other students to build an application from start to finish using the information we were learning in the class. My group decided to build Budget Buddy, an application that could help college students and young professionals budget their life and learn more about financial management.

APPLICATION DESCRIPTION

Budget Buddy is aimed at helping college students and young professionals budget their life. This application will include several features, including a spending and income log, a savings predictor, and an educational element to teach the young audience about investing and other financial management tasks.

Users will be able to easily log every expense they have into the software. They will be able to categorize their expenses to see where they spend the bulk of their money and where they might need to budget. They will also be able to log their income in the app to be able to notice if they are living above their means (spending more than they make) or not.

There will be several visualization tools such as tables and graphs that will help users visualize their spending habits and detect patterns for where they could reduce their spending. Users will then be able to implement a budget goal within the application (ex. “I will spend $200 a month on groceries”) and track their expenses to stay below their set budget. Users will also be able to set budgets for different periods of time (i.e. per month, per year, per semester, for summer, etc.).

Connected with the budget feature, there will be a savings predictor feature that allows users to see the amount of savings they will have if they stay within their newly set budget. This way, if a user has a certain savings goal, they will be able to set the correct budget to meet that goal.

The last feature of the application will be an educational feature. This part will hold information on how users will be able to start using more financial management tools and invest/save their money for their futures. This will include information on starting a user’s 401k, Roth IRA, investing in stocks, etc. There will be information in the application as well as links to other sites, tutorials, classes, and companies that can help users begin their financial journey.

BUILDING THE DATABASE:

When building out the database, we first brainstormed what data we wanted to store and be able to access. We came up with three main groups of data: Profile Data, Expenses Data, and Resources Data.

I mainly focused on the Expenses Data, because my main responsibility was the Expenses Page. The following data was included under this data grouping:

  • Primary Key: Transaction ID (unique to each specific expense entered into database)
  • Foriegn Key: Customer ID (links to Profile Data Grouping)
  • Transaction Type (ex. Expense or Cost)
  • Transaction Date (ex. 01-01-2022)
  • Category (ex. Groceries, Rent, Tuition, etc)
  • Amount (ex. $50)
Using .ejs files instead of .html files and SQL commands, I was able to add each new transaction a user entered into their specific database, and then automatically update the Expenses table to show the new expense.

USE CASE TESTING:

Feature 1: Create an Account
As a new user, I want to create an account, so I can access the website.
Acceptance Criteria
  • A user cannot create an account without filling in all of the mandatory fields:
    • First Name
    • Last Name
    • Email Address
    • Password
    • Confirm Password
  • Information is then stored in the 'users_db' table in the database

Feature 2: Login
As a current user, I want to be able to login to my account, so I can use the website.
Acceptance Criteria
  • User authentication fails when the user provides invalid credentials:
    • Username
    • Password
    • The credentials given by the user are then checked to see if it matches what is stored in the 'user_db' table in the database.
    • *Note that the password will be hashed before being inserted to the database.
    • They will then gain access to the rest of the website.
As a user, if the login credentials are incorrect, then I want an error message to appear.
Acceptance Criteria
  • The user isn't able to login if at least one of the login credentials doesn't match what is in the 'user_db' table.
    • Username
    • Password
    • The user doesn't get to bypass the login screen without the correct credentials.
    • The user will see a message that explains why they were not able to login.
Feature 3: View Expenses
As a user, I want to be able to view my expenses.
Acceptance Criteria
  • Once a user's login credentials are correct, the website will pull up the expenses of that person.
    • Information that is pulled matches the ID of the user that is currently logged in.
  • The data that corresponds to the current user will be pulled from the 'transactions_db' table. The data will be aggregated into the necessary expenses.
As a user, I want to be able to add recent transactions to my account.
Acceptance Criteria
  • A user cannot submit a transaction unless the mandatory fields are completed and with valid data types.
    • Name
    • Type
    • Date
    • Category
    • Amount
  • Once a user adds a transaction, then that information is stored in the 'transaction_db' table in the database.
Feature 4: Read/View Resources
As a user, I want to a page where financial information is available to learn
Acceptance Criteria
  • Once a user clicks on one of the resources, it will take them to the page of various resources to learn more about investment options.

REFLECTION:

I’ve never gotten the opportunity to create an application from start to finish, so this process was a great experience for me to get a glimpse into application development. While the UI and Front-End design still remain my favorite part of the process, I really enjoyed learning SQL and working with databases. I can now use this knowledge when I am making designs and thinking from the perspective of the engineers.

There were several obstacles that we had to overcome, such as one of our team members leaving the class halfway through the project. When this happened, we had to reevaluate the scope of our project and eliminate one of the features (the Predictions feature) in order to maintain feasibility.

Overall, this class was a great introduction in designing user interfaces, and led me to look for my first internship in UI/UX design!
